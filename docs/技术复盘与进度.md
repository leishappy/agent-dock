## 一、总体架构

```
┌──────────────────────┐                    ┌─────────────────────┐
│     PC (Python)       │                    │     ESP32-S3         │
│                        │    WiFi (WebSocket) │                      │
│  agentd.py ────────────┼──────────────────→│  ws_proto.cpp        │
│  (进程监控)             │  {"t":"agent",...} │  (JSON 协议解析)     │
│                        │                    │        ↓             │
│  浏览器 http://         │                    │  main.cpp 回调       │
│  agentdock.local ──────┼──────────────────→│  (串口打印 / UI渲染)  │
│  (Control Panel)       │  HTTP + WebSocket │                      │
└──────────────────────┘                    └─────────────────────┘
```

**核心设计决策**：

| 决策 | 选型 | 原因 |
|------|------|------|
| 框架 | Arduino（非 ESP-IDF） | 场景 IO 密集型，性能无差异；TFT_eSPI 库开箱即用 |
| 通信 | WiFi + WebSocket（非 USB CDC） | 无线的灵活性远超有线，配网后零配置 |
| 协议 | JSON 行（非二进制） | 人类可读，调试方便，ArduinoJson 已成熟 |
| 配网 | Captive Portal | 手机连热点自动弹页面，零门槛 |
| 服务发现 | mDNS | `agentdock.local` 比记 IP 地址优雅 |
| 前端 | 嵌入式 HTML（非 SPIFFS） | 当前页面量小，内嵌更简单；后续大批量静态资源再切 LittleFS |

## 二、ESP32 固件文件详解

### 2.1 `platformio.ini` — 项目骨架

```ini
platform = espressif32
board = esp32-s3-devkitc-1
framework = arduino

build_flags =
    -D ARDUINO_USB_CDC_ON_BOOT=1     # Serial 走 USB，开发调试用
    -D CONFIG_ASYNC_TCP_USE_WDT=0    # ESP32-S3 上 AsyncTCP 的 WDT 必须关

lib_deps =
    bodmer/TFT_eSPI@^2.5.43           # 显示驱动（ILI9341/ST7789）
    bblanchon/ArduinoJson@^7.0.4     # JSON 解析
    me-no-dev/ESPAsyncWebServer@^3.3.2  # 异步 HTTP + WebSocket
    me-no-dev/AsyncTCP@^3.2.7         # AsyncWebServer 的 TCP 层
```

**关键坑**：`CONFIG_ASYNC_TCP_USE_WDT=0` 不加的话 ESP32-S3 配网模式下会被看门狗踢死重启。

### 2.2 `src/User_Setup.h` — TFT_eSPI 配置

**这是踩坑最多的模块**。TFT_eSPI 的配置加载机制：

```
User_Setup_Select.h
  │
  ├── #ifndef USER_SETUP_LOADED   ← 如果此宏未定义
  │     #include <User_Setup.h>    ← 加载库自带的（里面是 ILI9341_DRIVER）
  │
  └── 然后根据 DRIVER 宏加载对应寄存器定义
       #if defined(ILI9341_DRIVER) → 加载 ILI9341 寄存器
       #elif defined(ST7789_DRIVER) → 加载 ST7789 寄存器
```

**解决方案**：在 `src/` 下创建自己的 `User_Setup.h`，定义 `USER_SETUP_LOADED` 阻止库自带配置，同时定义正确的 `ILI9341_DRIVER` 和所有引脚。PlatformIO 编译时 `src/` 的 include 优先级高于库目录，`#include <User_Setup.h>` 会找到我们的文件。

**当前配置**：
- 驱动：ILI9341
- 分辨率：240×320
- 引脚：CS=10, DC=9, RST=14, MOSI=11, SCLK=12, BL=2

### 2.3 `src/config.h` — 全局常量

所有「改一处全局生效」的配置集中在这里：
- 设备名、固件版本号
- WiFi 热点名、连接超时
- mDNS 主机名（agentdock.local）
- HTTP/WS 端口
- 屏幕尺寸
- NVS 存储 key 名
- 颜色主题（RGB565 调色板）

**设计模式**：`#define` 宏 + `constexpr` 命名空间混用。硬件相关用宏（编译期常量，不占 RAM），颜色用 constexpr namespace（有类型安全）。

### 2.4 `src/wifi_mgr.h/.cpp` — WiFi 管理器

**状态机**：
```
DISCONNECTED → CONNECTING → CONNECTED  (正常)
                           ↘ AP_MODE   (配网)
```

**核心方法**：

`begin()` — 启动决策：
1. 读 NVS 中保存的 WiFi 凭据
2. 有凭据 → `WiFi.begin(ssid, pass)` → 轮询 status 最多 15 秒
3. 连接成功 → 返回 true
4. 失败 → **自动清除错误凭据**（避免下次开机重试错密码）→ 进 AP 模式

`startAP()` — 配网热点：
- 创建开放热点 `AgentDock-Setup`
- 启动 DNS Server 劫持所有域名 → ESP32 IP（Captive Portal 关键）
- 手机连上后自动弹出配网页

`saveAndReboot()` — 保存并重启：
- 校验：SSID 不能空，密码不能 < 8 位（WPA2 要求）
- 写入 NVS → 延迟 → ESP.restart()

**配网页**：一个完整的单页应用（HTML + CSS + JS 全部内嵌在 C++ 字符串中）：
- 自动扫描附近 WiFi 列表
- SSID 输入 + 下拉选择
- 密码输入（8 位校验）
- 提交 → `/connect?ssid=...&pass=...` → 保存重启

**错误处理链**：
| 场景 | 行为 |
|------|------|
| 错误密码 | 15s 超时 → 状态码 4 → 清除凭据 → AP 模式 |
| SSID 不存在 | 状态码 1 → 清除凭据 → AP 模式 |
| 空 SSID 提交 | 前端拦截 + 后端拒绝 |
| 密码 < 8 位 | 前端提示 + 后端拒绝 |
| 首次开机 | NVS 无凭据 → 直接用 `isKey()` 检查，避免 error 日志 |

### 2.5 `src/ws_proto.h/.cpp` — 协议层

**角色**：JSON 消息的解析与分发，与传输层无关（USB CDC 和 WebSocket 共用同一套协议）。

**支持的消息类型**：

| type | 方向 | 示例 | 回调 |
|------|------|------|------|
| `tasks` | PC→ESP | `{"t":"tasks","list":[...]}` | onTasks |
| `weather` | PC→ESP | `{"t":"weather","temp":26,...}` | onWeather |
| `notify` | PC→ESP | `{"t":"notify","title":"...","level":"info"}` | onNotify |
| `agent` | PC→ESP | `{"t":"agent","status":"thinking","details":"..."}` | onAgent |
| `screen` | PC→ESP | `{"t":"screen","id":0}` | onScreen |
| `ping` | PC→ESP | `{"t":"ping"}` → 自动回复 pong | — |
| `hello` | ESP→PC | 握手时自动发送 | — |
| `ready` | ESP→PC | 客户端连接后发送 | — |

**设计模式**：
- **回调注册**：`proto.onXxx([](Data& d){ ... })` 注册业务回调
- **发送注入**：`setSendFn()` 让 WebServer 注入发送函数（因为 WS 发送需要 client 对象）
- **handleMessage()**：统一的入口，所有来源（WebSocket / HTTP API / 未来 BLE）都走这个函数

**内存策略**：
- 数据用栈上的 struct 和固定大小 char 数组（不动态分配）
- JsonDocument 使用默认大小（~256 bytes），够用
- 最多同时缓存 8 个任务

### 2.6 `src/web_server.h/.cpp` — HTTP + WebSocket 服务器

**架构**：基于 ESPAsyncWebServer，异步非阻塞。

**启动逻辑**：
```cpp
void begin() {
    setupCommonRoutes();        // /status, /forget（两种模式共用）
    if (AP 模式) setupAPRoutes();    // 配网页, /scan, /connect
    else         setupNormalRoutes(); // 控制面板, API, 404
    _ws->onEvent(onWsEvent);    // WebSocket 事件处理
    _http->begin();
}
```

**HTTP API 一览**：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/` | 配网页（AP）/ 控制面板（已连接） |
| GET | `/status` | JSON 设备状态 |
| GET | `/scan` | WiFi 扫描结果（AP 模式） |
| GET | `/connect?ssid=&pass=` | 保存 WiFi 并重启（AP 模式） |
| GET | `/forget` | 清除 WiFi 凭据并重启 |
| POST | `/api/tasks` | 发送任务数据 |
| POST | `/api/weather` | 发送天气数据 |
| POST | `/api/notify` | 发送通知数据 |
| POST | `/api/config` | 保存配置到 NVS |

**WebSocket 事件处理**（`onWsEvent`）：
```
WS_EVT_CONNECT:  注入 sendFn → sendHello → trigger onConnect 回调
WS_EVT_DATA:     解析 JSON → handleMessage → 触发业务回调
WS_EVT_DISCONNECT: 打印日志
```

**控制面板**（三 Tab SPA）：
- **Status Tab**：WiFi 状态、IP、WebSocket 客户端数、剩余内存
- **Test Tab**：任务/天气/通知的手动发送表单，调试用
- **Config Tab**：城市名、屏幕轮播间隔设置，存入 NVS

**API 统一处理**：`handleApi("tasks", req, data, len)` — 将 HTTP body 解析为 JSON，注入 `"t"` 字段，转交 `WsProto::handleMessage()`。这样网页提交和 WebSocket 消息走同一条通道。

### 2.7 `src/main.cpp` — 主调度

```cpp
setup():
  1. WiFi::begin() → 返回连接成功/失败
  2. 成功 → mDNS 注册 agentdock.local
  3. WebServer::begin()
  4. 注册所有 WsProto 回调 → 当前全部串口打印

loop():
  wifi.tick()    // DNS 请求处理（AP 模式）
  web.tick()     // WebSocket 客户端清理
  delay(20)      // 让出 CPU 给 AsyncTCP 任务
```

## 三、PC 端文件详解

### 3.1 `pc-app/agentd.py` — Claude Code 监控 Agent

**核心逻辑**：

```python
def detect_claude() -> dict:
    # 1. 用 psutil 遍历所有进程
    # 2. 匹配 "claude" / "cursor" / "node.*claude" / "anthropic"
    # 3. 通过 CPU 占用率 + 命令行参数推断状态:
    #    CPU > 15%              → thinking
    #    CPU > 5% + bash/shell  → tool
    #    CPU > 2%               → responding
    #    无匹配进程              → idle
```

**WebSocket 连接**：
- 连 ESP32 `ws://agentdock.local/ws`
- 每秒扫描一次进程状态
- 仅在状态变化时发送（避免重复推送）
- thinking/tool 状态额外发送一条 notify
- 断线自动重连（指数退避：2s → 3s → 4.5s → ... → 最多 30s）

### 3.2 `pc-app/ws_test.py` — WebSocket 测试客户端

一次性发送全套测试数据（tasks → weather → notify → screen 轮切 → ping），用于验证通信链路。
